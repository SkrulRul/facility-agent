# facility-agent

> Facility management agent API — debt facility agreements, covenants, repayment schedules, and default tracking for a facility agent administering agreements on behalf of one or more lenders.

## Stack

- **Python 3.13**
- **FastAPI** — async REST API
- **SQLAlchemy 2.0 (async) + asyncpg + Alembic** — Postgres persistence and migrations (production); in-memory dict repository (dev/test)
- **uv** — package management & virtual environment ([ADR-0022](docs/adr/0022-bootstrap-tooling-choices.md))
- **poethepoet** — task runner
- **mypy + pyright** — strict type checking
- **ruff** — linting & formatting
- **pytest** — testing

See [`CLAUDE.md`](CLAUDE.md) and [`docs/adr/`](docs/adr/) for the full architecture and decision record — every non-obvious choice in this codebase is documented there, not just asserted here.

## Setup

```bash
uv sync          # installs all groups (dev + test) automatically
```

## Run (local, no container)

```bash
uv run fastapi dev          # development (auto-reload)
uv run fastapi run          # production
```

API available at `http://127.0.0.1:8000`, docs at `http://127.0.0.1:8000/docs`.

By default, with no `DATABASE_URL` set, the API runs against an **in-memory repository** — data is lost on restart, no database required. This is the dev/test backend; it is never made production-capable ([ADR-0019](docs/adr/0019-database-backend-selection.md)).

## Database: Postgres + Alembic

Set `DATABASE_URL` to switch to the Postgres-backed repository:

```bash
export DATABASE_URL="postgresql+asyncpg://user:password@localhost:5432/facility_agent"
```

The URL **must** use the `postgresql+asyncpg://` scheme — a bare `postgresql://` fails validation at startup with an explicit error rather than a cryptic driver import failure ([`app/config.py`](app/config.py)).

Apply migrations before first use:

```bash
uv run alembic upgrade head
```

To generate a new migration after changing `app/repositories/orm_models.py`:

```bash
uv run alembic revision --autogenerate -m "describe the change"
```

Autogenerated migrations must be hand-reviewed before being trusted — the initial schema migration (`alembic/versions/14db6c81d4f0_initial_schema.py`) was hand-written (no live Postgres instance was available at the time), then run and verified end-to-end against a real Postgres instance during this Docker packaging phase, confirming it matches `app/repositories/orm_models.py`. See [ADR-0021](docs/adr/0021-agreement-repository-update.md) for the migration-review rationale.

## Run in Docker

A single, API-only `Dockerfile` builds and runs the API — it does **not** bundle or orchestrate Postgres. Point it at your own instance via `DATABASE_URL`; this is a deliberate scope boundary, not a shortcut — see [ADR-0023](docs/adr/0023-single-container-docker-packaging.md).

```bash
docker build -t facility-agent .

# Against your own Postgres — migrations run automatically on container start:
docker run --rm -p 8000:8000 -e DATABASE_URL="postgresql+asyncpg://user:password@host.docker.internal:5432/facility_agent" facility-agent

# Or with no database at all — falls back to the in-memory backend:
docker run --rm -p 8000:8000 facility-agent
```

The container's entrypoint ([`docker-entrypoint.sh`](docker-entrypoint.sh)) runs `alembic upgrade head` automatically, but only when `DATABASE_URL` is set — there's no schema to migrate against the in-memory backend.

## CI

GitHub Actions ([`.github/workflows/ci.yml`](.github/workflows/ci.yml)) runs on every push and pull request to `main`, executing the same gate you'd run locally:

```bash
uv run poe check     # lint + typecheck + test — identical to CI
```

CI has **no Postgres service** and does not run the Postgres-backed test suite or any of the manual smoke-test scripts (`scripts/smoke_test_*.py`) — those stay manual by design, consistent with how every other real-I/O boundary in this codebase (LLM extraction, MCP server, persistence) is verified. See [ADR-0024](docs/adr/0024-ci-scope-lint-typecheck-in-memory-tests.md).

## Authentication

Every `/agreements` route requires an `X-API-Key` header. Each key is mapped to exactly one role via env vars — comma-separated for multiple keys per role:

```bash
export LOAN_OPERATIONS_ANALYST_API_KEYS="analyst-key-1,analyst-key-2"
export CREDIT_RISK_OFFICER_API_KEYS="risk-officer-key-1"
```

A missing or unrecognized key returns `401`. A recognized key for the wrong role (Credit Risk Officer calling a write endpoint) returns `403`. `GET /health` is unauthenticated by design (infra liveness probe). See [ADR-0026](docs/adr/0026-role-based-api-key-authentication.md) and [`docs/specs/auth.md`](docs/specs/auth.md).

## MCP Server

The service layer is also exposed as an MCP server (`app/mcp_server.py`) — a separate, independent entry point over stdio, not mounted on the FastAPI app. It exposes two read-only query tools: `get_agreement(agreement_id: UUID) -> FacilityAgreement` and `list_continuing_defaults(agreement_id: UUID) -> list[DefaultEvent]`. See [ADR-0018](docs/adr/0018-mcp-server-fastmcp-stdio.md) and [`docs/specs/mcp_server.md`](docs/specs/mcp_server.md) for the full design and a known limitation around per-process in-memory state.

```bash
uv run python -m app.mcp_server    # runs the MCP server over stdio
```

To point a Claude Code MCP client at it, add an entry to `.mcp.json` (or your client's config) with `command: "uv"`, `args: ["run", "python", "-m", "app.mcp_server"]`, and `cwd` set to this repo's root.

If `DATABASE_URL` is set, the MCP server and the HTTP API share the same live database — cross-process data becomes visible to both, a direct consequence of backend selection ([ADR-0019](docs/adr/0019-database-backend-selection.md)).
